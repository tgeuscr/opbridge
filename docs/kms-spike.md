# Heptad KMS Compatibility Spike

Goal: prove AWS KMS can replace local relayer signing for both Heptad relay paths.

## What this spike proves

- OPNet path:
  - AWS KMS `ML_DSA_44` can sign the exact bytes Heptad relay code signs
  - returned public key length matches bridge expectation
  - returned signature length matches bridge expectation
  - AWS KMS verify succeeds on the same message

- Ethereum path:
  - AWS KMS `ECC_SECG_P256K1` can sign the exact digest the vault expects
  - DER signature can be converted into packed `r || s || v`
  - recovered signer address matches the KMS public key
  - AWS KMS verify succeeds on the same digest

This does not fully prove production readiness until a live attestation is accepted by the deployed bridge and vault.

## KMS keys to create

OPNet relay key:

- `KeySpec=ML_DSA_44`
- `KeyUsage=SIGN_VERIFY`

Ethereum relay key:

- `KeySpec=ECC_SECG_P256K1`
- `KeyUsage=SIGN_VERIFY`

## OPNet spike

Use either a direct message hex blob or a Heptad mint attestation JSON object.

Direct message bytes:

```bash
KMS_KEY_ID=<opnet-kms-key-id> \
OPNET_MESSAGE_HEX=0x<32-byte-attestation-hash-or-message-bytes> \
npm run kms:spike:opnet --workspace @heptad/relayer
```

Heptad mint attestation JSON:

```bash
KMS_KEY_ID=<opnet-kms-key-id> \
OPNET_ATTESTATION_JSON='{
  "version": 1,
  "ethereumVault": "0x1234...",
  "opnetBridgeHex": "0xabcd...",
  "ethereumUser": "0x5678...",
  "opnetUser": "0x90ab...",
  "assetId": 0,
  "amount": "1000000",
  "direction": 1,
  "nonce": "42"
}' \
npm run kms:spike:opnet --workspace @heptad/relayer
```

Expected success conditions:

- `publicKeyBytes = 1312`
- `signatureBytes = 2420`
- `kmsVerify = true`
- `ok = true`

## Ethereum spike

Use either a direct 32-byte digest or a Heptad release attestation JSON object.

Direct digest:

```bash
KMS_KEY_ID=<evm-kms-key-id> \
EVM_MESSAGE_DIGEST_HEX=0x<32-byte-release-attestation-hash> \
npm run kms:spike:evm --workspace @heptad/relayer
```

Heptad release attestation JSON:

```bash
KMS_KEY_ID=<evm-kms-key-id> \
EVM_RELEASE_JSON='{
  "version": 1,
  "ethereumVault": "0x1234...",
  "opnetBridgeHex": "0xabcd...",
  "ethereumUser": "0x0000000000000000000000005678...",
  "opnetUser": "0x90ab...",
  "assetId": 0,
  "amount": "1000000",
  "direction": 2,
  "nonce": "7"
}' \
npm run kms:spike:evm --workspace @heptad/relayer
```

Expected success conditions:

- `packedSignatureBytes = 65`
- `recoveredAddress == ethereumAddress`
- `kmsVerify = true`
- `ok = true`

## Decision rule

Use KMS for mainnet only if:

1. both spike scripts succeed
2. one live OPNet attestation generated by KMS is accepted by the deployed bridge
3. one live Ethereum release signature generated by KMS is accepted by the vault
4. latency and retry behavior are acceptable on EC2
